# Caduceus

A Hospital Information System (HIS) demo that exercises every published `@fhir-dsl/*` package — built on TanStack Start with shadcn UI and React Query.

This app is the canonical end-to-end consumer of the `@fhir-dsl/*` surface freeze. It doubles as a smoke test, a marketing screenshot source, and a place to flush out remaining sharp edges in the consumer-facing API.

## What it demonstrates

| Pillar | Package(s) | Where in the app |
|---|---|---|
| Typed search builder + post-fetch projection | `core`, `runtime`, `fhirpath` | `/patients` and `/patients/$id` |
| FHIRPath builder, evaluator, UCUM, write-back, terminology hooks | `fhirpath` | `/playground` |
| SMART-on-FHIR v2 patient launch (PKCE) | `smart` | `/launch` → `/callback` → `/patients` |
| `FhirDslError` + `Result<T, E>` toolkit | `utils` | `lib/error-toast.ts` (every error path) |
| MCP server bridge for an LLM agent | `mcp` | `src/server/mcp-singleton.ts` + `/api/mcp` + `/api/mcp/audit` |
| Claude chatbot wired to the MCP server | `mcp` + Anthropic SDK | `/api/chat` server route + `components/chatbot/chat-drawer.tsx` |
| Code generation from R4 | `cli`, `generator`, `terminology` | `pnpm gen` produces `src/fhir/r4/` |

## Quickstart

```bash
# Install workspace deps (from repo root).
pnpm install

# Generate R4 types (downloads HL7 spec into .fhir-cache/).
pnpm --filter @fhir-dsl/his-demo run gen

# Run the dev server.
pnpm demo
# → http://localhost:3000
```

Or from the repo root: `pnpm demo:gen` + `pnpm demo`.

### Environment

Copy `.env.example` to `.env` and set whichever variables apply:

| Variable | Purpose |
|---|---|
| `ANTHROPIC_API_KEY` | Required for the chatbot. Stays on the server (read inside `api/chat.ts`). |
| `FHIR_BASE_URL` | Upstream FHIR server. Default: `https://hapi.fhir.org/baseR4`. |
| `FHIR_BEARER_TOKEN` | Optional bearer for the MCP server's outbound calls. |
| `VITE_SMART_LAUNCHER_URL` | Smart Health IT launcher URL. Default: `https://launch.smarthealthit.org`. |
| `VITE_SMART_LAUNCHER_FHIR_VERSION` | FHIR version the launcher should simulate. Default: `r4`. |
| `VITE_SMART_CLIENT_ID` | OAuth client id (public). Default: `fhir-dsl-demo`. |
| `VITE_SMART_REDIRECT_URI` | Where the IdP redirects back. Default: `http://localhost:3000/callback`. |
| `VITE_SMART_APP_LAUNCH_URL` | Our /launch route — the launcher sends users here with `?iss=…&launch=…`. Default: `http://localhost:3000/launch`. |

`VITE_*` variables are exposed to the browser; everything else stays server-side.

## Known caveats

1. **SMART entry must go through the launcher.** Smart Health IT's `/v/r4/fhir` endpoint is not a standalone-launch target — it requires a real `launch` token. The "Login with SMART" button takes users to the SHI launcher first; the launcher then redirects back to our `/launch` with `?iss=…&launch=…` populated. If you bookmark `/launch` directly with no `iss`, we forward you to the launcher.
2. **R4 throughout.** Demo generates R4 types and queries an R4 HAPI server, matching the Smart Health IT sandbox. To bump to R5 once the sandbox catches up, change the `--version` flag in the gen script and update import paths.
3. **No emitted validators yet.** `pnpm gen` runs without `--validator native` because the generator's native emitter trips on self-referential backbone elements (`CodeSystem.concept`, `Composition.section`, etc.) — TDZ violations at module load + a fistful of TS errors. Tracked. Demo still demonstrates `compileInvariant` / `validateInvariants` directly through the playground.
4. **MCP server in-process.** The dispatcher singleton lives inside the Nitro process. For deployments where the MCP needs to be reachable from outside, swap the in-process call sites in `api/chat.ts` for HTTP calls against `httpTransport()` mounted on a separate port.

## Verification flow

1. `pnpm install && pnpm demo:gen && pnpm demo` — site loads at <http://localhost:3000>.
2. Click **Login with SMART** — bounce through the sandbox → land on `/patients` with launch-context card visible.
3. Open a patient → vitals tab projects BP via FHIRPath; write-back tab shows the JSON Patch preview.
4. Open `/playground`, click a snippet, hit **Run** → see `compile()` + `evaluate()` output. Switch to the UCUM-Celsius snippet to see `UcumError` surface with its `kind`.
5. Open the chat drawer (header **Chat** button), ask "list 3 patients named Smith". Observe streaming reply + tool-call timeline + audit panel ticking.
6. Force a 404 (`/patients/does-not-exist`) — ErrorBlock shows `core.request` with the typed `context.status`.
